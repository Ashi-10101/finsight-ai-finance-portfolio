金策 FinSight 技术架构与业务流程图集

版本：v1.0 ｜ 依据：00_金策_FinSight_项目事实库.md 状态边界：金策 FinSight 当前处于方案设计 + 产品原型阶段，未开发完成、未上线。以下 8 张 Mermaid 图记录目标架构和接口边界，不是现网拓扑。Mindraft AI Gateway 已实现的 Pipeline MoA、SSE trace、runs/run_steps 落库和模型路由只作为独立先导工程依据；金融数据治理、证据链、组合适配、合规审批及金仕达集成仍属金策 MVP 拟实现或后续拟接入范围。

图 1 产品架构图（六层架构 + 用户层 + 输出层）

技术分层与七大产品模块是两个不同维度的划分：七大模块描述用户在产品中完成的工作，六层架构描述这些能力在系统中的技术承载与依赖关系。

这张图用于固定产品模块边界，而不是表示各层已经存在。证据链单独位于分析引擎之后，是因为原始检索片段不能直接当作结论依据；组合与客户材料必须消费经过来源、时间和推断类型标记的中间结果。合规层画成横向约束，原因是适当性、权限和审计不能只在最终文案上补做。

MVP 先收敛到公开信息检索、研究草稿、证据字段、模拟组合计算和人工确认。知识图谱、完整客户画像接入、自动多方案比较及面向真实客户的输出暂不实现；客户沟通材料仅保留演示模板。

图 2 技术架构图

工程上把编排层与模型网关拆开：前者负责任务状态、回退和人工节点，后者只处理模型供应商、角色路由和调用记录。这样更换模型时不改业务状态机，也能独立核算每个阶段的成本。Mindraft AI Gateway 已有四阶段 Pipeline、SSE trace、runs/run_steps 落库和统一模型接入，可复用其工程模式；这些代码尚未等同于图中的金策服务。

私有化边界是目标部署约束，不代表当前已有机构内网环境。MVP 可先采用向量检索加结构化查询，知识图谱检索、完整 RBAC、机构级数据分级和自动降级策略待接口与数据条件明确后再做，避免同时维护三套不成熟索引。

图 3 数据流程图

数据管线优先保存来源、发布时间、更新时间和原文位置，因为这些字段事后通常无法从向量片段可靠恢复。冲突检测不自动选择“正确版本”，而是把口径、时间或来源冲突交给上层标注和人工复核；否则清洗阶段的静默覆盖会让报告无法解释。

MVP 计划先接公开公告、政策材料和少量结构化示例数据，建立可复现的语料快照。新闻、研报、内部材料仅在授权和权限模型确定后接入；知识图谱、全量实体消歧和自动失效传播暂缓。向量库与结构化库先满足检索和重算，图谱库保留接口但不作为首版依赖。

图 4 多智能体协作图

11 个角色是责任划分，不要求 MVP 启动 11 个独立模型实例。首版可由同一基础模型按角色提示词串行执行，角色输出使用不同 schema，以便定位事实、因果、组合或合规环节的错误。人工确认放在客户适配之后，是为了阻止未经确认的具体处置表述继续进入合规和整合阶段。

Mindraft AI Gateway 已有 Planner → Worker → Reviewer → Integrator 和单轮 repair，可作为编排起点。并行 Worker、角色/模型编辑器和复杂冲突仲裁在该先导项目中仍属规划中，因此金策 MVP 不假定并行吞吐已经实现；先验证串行流程、回退上限和日志可追溯性。

图 5 证据链生成流程图

证据链采用“先定位原文，再允许生成结论”的顺序，目的是让引用覆盖和引用支持可以分别计分。推断字段不能伪装成来源原话；假设、反例和失效条件与结论一并保存，便于数据更新后判断哪些段落需要重审。

MVP 不尝试自动证明因果，也不把模型输出的置信度当作统计概率。首版先实现证据编号、时间、原文锚点、陈述类型和人工确认状态；自动反例检索、置信度校准、失效条件触发和跨报告依赖更新待评测数据积累后再实现。

图 6 用户业务流程图

业务流把内部研究报告和客户材料放在同一证据版本上，但两者权限和措辞规则不同。投资经理节点只接收研究参考，不接收自动交易指令；客户交付必须晚于研究员确认与合规审批。这样可以在同一 run 中追踪“哪条证据经过谁确认后进入哪份材料”。

MVP 演示到“内部草稿 + 模拟客户材料 + 审批占位状态”为止，不连接真实客户、不执行发送，也不产生交易指令。跨角色账号、电子签批、机构档案系统和不可篡改归档需结合实际部署环境，暂不在原型中承诺。

图 7 合规审核流程图

审核顺序先处理可确定的硬规则，再处理需要语义判断的敏感建议，便于解释每次阻断原因。适当性检查依赖客户风险等级、期限和流动性等结构化字段；字段缺失时应返回“无法判断”，不能默认通过。规则命中和模型判断都只形成预审结果，最终签发仍由具备权限的人员完成。

MVP 先做收益承诺词、风险提示字段、证据绑定和 AI 标识检查，并记录规则版本。真实适当性引擎、机构规则库、审批权限、电子签名和归档策略尚未实现；图中的“通过”表示目标状态迁移，不表示当前样稿已经获得合规批准。

图 8 金仕达系统集成示意图

该图只定义潜在集成面，不假定金仕达具体系统的接口、数据模型或部署方式。优先采用 API 和事件消息，是为了把研究任务、证据版本和审批状态与交易核心解耦；统一认证和审计应由企业侧策略约束，金策不自行扩大数据权限。

当前没有接入金仕达任何生产系统。MVP 不实现交易指令回传，最多以 mock 接口演示任务输入和研究结果输出。行情、风控、合规、客户、财富管理、CRM、权限审计及内部研究资料均为后续拟接入项，只有在企业授权、接口确认和安全评审后才能进入开发。
