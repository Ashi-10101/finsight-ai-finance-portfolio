# 金策 FinSight 飞书实现方案

> 当前状态：方案设计 + 产品原型阶段，未开发完成、未上线。本文档涉及的飞书产品能力、可用性、配额与 API 细节**均未实测**，全部标注「待实测确认」；团队尚未在飞书上实际搭建任何对象。本文是 `docs/01_开题方案.md`、`docs/05_技术架构与业务流程.md`、`docs/06_评测方案.md`、`docs/08_产品与客户数据模型.md` 引用的事实基础。

## 引言：这份文档要回答什么

**核心判断**：金策是一套借助飞书 AI 工具、面向中小 B 与大 C 的智能化金融投研与财富助手。要把这个定位落成可搭建的方案，必须回答三个问题——为什么是飞书、飞书承载什么、飞书做不到的部分由什么补。这份文档只回答这三件事。

命题方的要求是：借助飞书 AI 工具，探索面向中小 B 与大 C 的智能化金融投研与财富助手，覆盖资讯解读、研报摘要、市场观点整理、产品理解、客户画像、风险提示与服务话术生成。

## 一、为什么是飞书

**核心判断**：中小 B（区域券商分支、独立财富工作室、三方销售、保险经代、家族办公室、私募 FOF）与大 C（高净值与资深个人投资者）没有 IT 部门、没有投研团队、也没有 IT 预算；飞书是他们中相当比例机构**唯一已经在用的"系统"**【推断：基于对中小金融机构办公工具的普遍观察，未做正式用户访谈】。因此飞书是唯一能**零部署**进入中小 B 的路径。

### 1.1 中小 B 的真实处境

| 约束 | 对工具选型的含义 |
|---|---|
| 没有 IT 部门，没人能维护私有化部署 | 工具必须是"装上就能用"的 SaaS，不能要求运维 |
| 没有专职投研团队，研究由顾问本人兼职完成 | 工具必须直接产出可用的话术与摘要，而不是再给一套研究框架 |
| 没有 IT 预算，无法采购行情终端级别的系统 | 工具必须依托已有的办公软件，边际成本接近零 |
| 客户在微信、飞书群里，不在机构自有 App 里 | 工具的输出必须能进入即时通讯场景，而不是停留在独立工作台 |

### 1.2 数据口径：数据不出机构飞书租户

金策的数据原则是：**数据不出机构自己的飞书租户**。具体含义是：

- 客户库、产品库、观点库等结构化数据，存放在机构自己开通的飞书租户内的多维表格中，权限由机构管理员控制；
- 不将客户数据导入任何第三方的独立系统；
- 自研外挂服务（见第三章）若必须处理客户字段，需单独论证其数据流向，不能默认豁免"不出租户"的承诺（详见 §3.3 的如实讨论）。

飞书环境已确认为**企业版**，记录级权限、字段级权限与管理后台数据隔离在能力范围内（具体套餐与可用性待实测确认）。

### 1.3 平台边界与"零部署"的精确含义

依托飞书平台需要接受三个约束，如实记录：其一，扩展方式受平台约束——数据模型迁就多维表格的字段类型，编排迁就 AI 工作流与智能伙伴创建平台的节点能力，界面迁就消息卡片与侧边栏的交互范式；其二，存在供应商锁定风险——数据、流程与话术模板一旦沉淀在飞书租户内，迁移到其他平台的成本会很高【推断】；其三，要求数据进入自有内网的私有化场景不在本方案覆盖范围内。

还需精确界定"零部署"这一承诺的边界，它并非无条件成立：**飞书侧**（多维表格、知识问答、AI 字段捷径、AI 工作流、审批流、群机器人）对机构零部署；**外挂侧**（第三章的四项自研能力）是否零部署，取决于外挂服务的部署形态——在 §3.3 的形态三（金策统一托管、只处理已脱敏数据）下对机构零部署，在形态一（机构自托管）下则需要机构具备基本的部署能力。本章"零部署进入中小 B"的论证，以 MVP 默认采用形态三为前提。

## 二、系统在飞书上的承载

**核心判断**：金策的六项系统职责——数据、检索、编排、应用、合规、部署——全部由飞书原生组件承载；承载之后仍然存在的能力缺口，才是第三章要处理的问题。

### 2.1 承载总表

| 系统职责 | 飞书承载 | 承载状态 |
|---|---|---|
| 数据层 | 飞书多维表格四张表：资讯研报库 / 观点库 / 产品库 / 客户库 | MVP 计划做，待搭建（字段设计已完成，见 `docs/08`） |
| 检索层 | 飞书知识问答（挂产品说明书与研报） | MVP 计划做，待实测确认 |
| 编排层 | 飞书多维表格 AI 字段捷径 + 飞书 AI 工作流；复杂编排用飞书智能伙伴创建平台（Aily） | MVP 计划做，待实测确认 |
| 应用层 | 群内 @飞书群机器人 + 飞书消息卡片 + 飞书侧边栏应用 | MVP 计划做，待实测确认 |
| 合规层 | 飞书审批（审批流） | MVP 计划做，待实测确认 |
| 部署 | 数据留在机构自己的飞书租户内 | 原则已确立，技术细节待实测确认 |

### 2.2 数据层：四张多维表格

**承载什么**：资讯研报库（原文链接、发布时间、入库时间、AI 摘要）、观点库（结论、立场、适用情景、失效条件）、产品库（产品说明书要点、风险等级、费率、适当性要求）、客户库（脱敏后的画像、风险等级、期限、流动性约束）。四张表把"资讯→观点→产品→客户"这条链路固化成数据结构，字段级设计见 `docs/08_产品与客户数据模型.md`。

**为什么是它**：多维表格是飞书内唯一同时具备结构化存储、字段级权限、AI 字段捷径与自动化触发能力的载体。

**MVP 首版做到哪一步**：建立四张表的字段设计，导入少量公开资讯与模拟客户数据，验证 AI 字段捷径能否完成摘要与标签化。

**暂不实现**：实时行情接入、全量研报入库、跨租户的表间关联。

### 2.3 检索层：知识问答

**承载什么**：把产品说明书、研报原文等长文档挂载到飞书知识问答，供顾问在群或侧边栏中直接提问。

**为什么是它**：中小 B 场景下，"语义检索"这一层即可覆盖绝大多数查询需求；知识问答是飞书原生提供这一层能力的组件，机构无需维护任何索引。

**MVP 首版做到哪一步**：挂载若干份产品说明书与公开研报，验证问答的引用回溯能力。

**暂不实现**：混合检索、图谱查询、检索结果的字段级锚点输出（锚点问题见 §3.2）。

### 2.4 编排层：AI 字段捷径 + AI 工作流 + 智能伙伴创建平台

**承载什么**：AI 字段捷径负责"单条记录内的文本变换"（摘要、分类、标签、话术改写）；AI 工作流负责"跨记录、跨表、带触发条件的流程"（新资讯入库→生成观点→匹配产品→匹配客户→生成话术）；智能伙伴创建平台（Aily）负责更复杂的多轮对话式编排。

**为什么是它**：这三者覆盖"任务规划→执行→整合"的完整管线，且全部在飞书租户内运行，不需要自建后端。

**MVP 首版做到哪一步**：用 AI 字段捷径完成资讯摘要与观点标签，用一条 AI 工作流跑通"资讯→观点→话术"的单条链路。

**暂不实现**：多智能体并行编排、复杂冲突仲裁、跨工作流的状态共享。

### 2.5 应用层：群机器人 + 消息卡片 + 侧边栏应用

**承载什么**：群内 @机器人触发查询与生成；消息卡片承载结构化的观点与话术输出（含确认按钮）；侧边栏应用承载需要持续查看的工作区（如客户画像面板）。

**为什么是它**：中小 B 的工作场景就在群里，输出必须进入群而不是独立工作台；消息卡片是飞书内唯一能同时承载结构化内容与交互按钮的载体。

**MVP 首版做到哪一步**：配置一个群机器人，能用消息卡片输出观点与话术草稿。

**暂不实现**：自定义侧边栏应用的完整开发、客户侧的直接触达。

### 2.6 合规层：飞书审批流

**承载什么**：每一条话术对外发出前的人工确认节点，由飞书审批流承载。

**为什么是它**：这是本章的关键论点。飞书审批流**天然带有责任人、时间戳与不可抵赖的审批记录**——每一条话术对外发出前，必须经过具名人员的"同意"或"驳回"，记录自动留存，不可事后修改【推断：基于审批流的一般产品逻辑，具体防篡改能力待实测确认】。对金融场景而言，这种留痕能力直接满足"操作可审计、责任可追溯"的要求（参见 `docs/01_开题方案.md` §十六），且机构无需为此开发任何功能。换句话说：**合规这一层，飞书原生能力直接可用，是六项职责中缺口最小的一层。**

**MVP 首版做到哪一步**：建立一条"话术发布前必须审批"的审批流，验证审批记录能否与具体话术版本关联。

**暂不实现**：审批流与客户适当性检查的自动联动、审批记录的独立归档导出。

## 三、飞书原生能力的边界：我们必须自己补什么

**核心判断**：飞书原生 AI 擅长**无状态的文本变换**——摘要、分类、抽取、翻译、改写；不擅长需要**跨记录一致性、字段级锚点、时间维度上的重审、可解释拦截理由**的任务。金策的可信性要求（证据绑定、失效重审、引用校验、违规拦截）恰好全部落在后者，因此必须以外挂形式自研补齐。

### 3.1 边界判定方法

判断一项能力"飞书原生能否直接做"，用以下标准：

| 维度 | 飞书原生 AI 能直接做 | 飞书原生 AI 不能直接做 |
|---|---|---|
| 状态 | 无状态：单次输入→单次输出 | 有状态：需要记住"这条结论依赖哪些数据、数据更新后要重审" |
| 锚点 | 输出文本本身 | 输出文本 + 原文锚点（来源、段落位置、发布时间） |
| 一致性 | 单条记录内自洽 | 跨记录、跨表的一致性（如话术与观点库结论一致） |
| 拦截 | 给出判断结果 | 给出**可解释的**拦截理由（命中哪条规则、为什么） |

凡落在右列的需求，一律不假定飞书原生能力可以满足，一律进入外挂清单。下面四项外挂均由这条标准推出。

### 3.2 四项自研外挂

#### 外挂一：证据字段级绑定

**解决什么问题**：每一条 AI 生成的观点与话术，必须绑定到原始材料的来源、段落位置与发布时间（对应 `docs/01_开题方案.md` §十三 的九字段证据要求中的第 1–3 项）。

**飞书为什么做不到**：AI 字段捷径的输出是文本，不输出原文锚点；知识问答能给出引用，但引用粒度与"字段级绑定到具体记录的具体段落"不是同一回事【待实测确认】。没有锚点，复核者无法核对"这句话的出处是哪份材料的哪一段"，证据链在第一步就断了。

**打算怎么实现**：资讯研报库增设"来源链接 / 发布时间 / 原文段落"字段；AI 生成观点时，外挂服务要求模型在输出中携带锚点标记，写回观点库的对应字段；锚点缺失的观点自动标记为"待人工补锚"。

**如何验证**：对照实验中统计"带有效锚点的观点比例"与"锚点可回溯率"，MVP 目标值（待验证）：锚点字段完整率 = 100%，锚点可人工回溯率 ≥ 95%。

#### 外挂二：失效条件登记与到期重审触发

**解决什么问题**：每条结论成立的前提（失效条件）要在相关数据更新后被重新检查，而不是生成后永远有效（对应 `docs/01_开题方案.md` §十三 第 9 项与 §十四 的失效条件论证）。

**飞书为什么做不到**：AI 字段与工作流的触发是"记录变更→执行一次"，不具备"登记一个条件、在条件被触发时回头重审历史结论"的机制；这是时间维度上的状态管理，属于 3.1 表中"有状态"一类【待实测确认】。

**打算怎么实现**：观点库增设"失效条件 / 失效状态 / 最后重审时间"字段；外挂服务监听资讯研报库与行情类字段的更新，命中某条观点的失效条件时，将该观点标记为"待重审"并通知责任顾问。

**如何验证**：预埋"数据更新使旧结论失效"的测试用例，统计到期重审的触发召回率，MVP 目标值（待验证）：召回率 ≥ 90%，无误触发率 ≥ 85%。

#### 外挂三：引用支持率自动校验

**解决什么问题**：检查"引用是否真的支持了这条结论"，而非只检查"有没有引用"（对应 `docs/06_评测方案.md` §4.1 的引用支持率指标）。

**飞书为什么做不到**：知识问答与 AI 字段只保证"给出引用"，不保证"引用在语义上支持结论"；FinanceBench 的经验（GPT-4-Turbo + RAG 约 81% 答错或拒答，arXiv:2311.11944）表明，取证失败恰恰是最常见的失效模式——引用了看似相关、实则不支持结论的段落【学术研究】。

**打算怎么实现**：外挂服务对每条"结论 + 引用"对做二次校验：抽取引用段落，让独立模型判断"支持 / 部分支持 / 不支持"，判定规则沿用 `docs/06_评测方案.md` §4.3 的细则（部分支持仅用于"方向支持但强度、范围或时间限定不支持"的情况）；判定为"不支持"的结论自动降级或标记。

**如何验证**：对照实验中按 `docs/06_评测方案.md` §4.1 的口径统计引用支持率，MVP 目标值（待验证）：引用支持率 ≥ 90%，无依据结论比例 ≤ 5%（红线）。

#### 外挂四：话术违规表述拦截

**解决什么问题**：拦截收益承诺、保本、稳赚、无依据建议四类高风险表述，且拦截必须给出可解释的理由（命中哪条规则、为什么拦截）。

**飞书为什么做不到**：飞书原生能力没有内置金融合规词表与适当性规则；即便用提示词约束模型自查，模型也只能给出"是否违规"的判断，无法稳定给出"命中规则编号 + 命中片段 + 处置建议"的结构化拦截理由——而金策的审核原则是"每一次阻断都能给出可解释的理由"（见 `docs/05_技术架构与业务流程.md` 图 7）。

**打算怎么实现**：外挂服务内置四类规则表（收益承诺 / 保本稳赚 / 无依据建议 / 适当性冲突），话术在写入"待审批"状态前必须先过拦截器；命中的话术附带拦截理由进入审批流，审批人能看到"系统为什么拦"。

**如何验证**：对抗样本测试，MVP 目标值（待验证）：四类预埋违规表述拦截率 = 100%（红线），误拦率 ≤ 10%。

### 3.3 外挂的实现形态

**实现形态**：飞书多维表格自定义字段（扩展字段类型）+ Webhook 外挂服务。数据流向与调用时机如下：

| 环节 | 数据流向 | 调用时机 |
|---|---|---|
| 观点生成 | 资讯研报库记录 → Webhook → 外挂服务 → 写回观点库（含锚点） | 新资讯入库或手动触发 |
| 失效重审 | 数据更新 → Webhook → 外挂服务 → 更新观点库失效状态 | 监听字段变更 |
| 引用校验 | 观点库记录 → Webhook → 外挂服务 → 写回校验结果字段 | 观点生成后立即执行 |
| 违规拦截 | 话术草稿 → Webhook → 外挂服务 → 写回拦截结果与理由 | 进入审批流之前 |

权限要求：外挂服务需要读写四张多维表格的权限，权限范围按最小必要原则配置【具体授权机制待实测确认】。

**关于"数据不出租户"承诺的如实讨论**：外挂服务本身部署在哪里，直接决定上述承诺是否成立。三种形态：

1. **外挂服务部署在机构自有环境**（如机构自己的服务器或云账号）：数据流出飞书租户、进入机构自有环境，承诺的口径从"不出飞书租户"调整为"不出机构控制域"，承诺仍然成立，但要求机构具备最低限度的部署能力，这对部分中小 B 是负担【推断】。
2. **外挂服务部署在第三方云**（与机构无委托关系的通用云服务）：数据离开机构控制域，承诺不成立，不可接受。
3. **外挂服务由金策统一托管，但只处理已脱敏数据**：机构无需部署任何东西，外挂服务以 SaaS 形式提供。这一形态成立的理由需要拆开说。其一，按 §4.1 的脱敏规则，客户姓名、手机号、身份证号不入表，持仓金额区间化、持仓明细类别化；因此外挂服务实际经手的数据只有资讯原文、观点结论、话术草稿与区间化画像，本身不含个人信息——其中话术草稿以占位符形式存储客户称呼，不含个人称谓（详见 `docs/08_产品与客户数据模型.md` §六）。其二，"数据不出机构控制域"这一承诺真正要保护的是个人信息与机构敏感数据——既然个人信息从未进入多维表格，也就不存在"流出"的问题。其三，非个人信息的受托处理边界，按《证券期货业网络和信息安全管理办法》（证监会令第 218 号）中关于委托处理的要求，由金策与机构以协议明确约定；该法规已列入 §4.3 的对应表，此处的委托处理安排正是该表"委托处理边界"一栏在外挂场景下的具体落实。其四，必须诚实交代这一形态的前提：它成立的**唯一条件**是 §4.1 的脱敏规则被严格执行——一旦有机构为了追求匹配精度，把真实姓名或精确持仓金额写进客户库，上述全部论证立即失效。因此 §4.1 不是可选的最佳实践，而是形态三的前置条件。

因此 MVP 的计划是：**默认采用形态三**——外挂服务由金策统一托管，对中小 B 保持零部署，与第一章的立论一致；同时保留形态一作为对数据管控要求更高的机构的可选项。形态二不可接受的判断维持不变。演示阶段外挂服务部署在团队自有服务器上，并在文档中向用户明示数据流向与委托处理约定。这不是一个可以回避的问题，必须如实交代。

### 3.4 与"飞书原生直接做"的对照实验

评测方案中设有一个对照组：**D 组 = 飞书原生 AI 直接做**——仅用 AI 字段摘要 + 知识问答，不加任何自研外挂（设计见 `docs/06_评测方案.md` §2）。

这一组的意义是回答一个无法回避的质疑：**"你做的这些，飞书自己不就能做吗？"** 没有 D 组，四项外挂的价值就无法归因——无法证明证据绑定、失效重审、引用校验与违规拦截带来的增益，是外挂产生的，还是飞书原生能力本来就有。D 组与 C 组（金策方案）共享同一批任务、同一份语料与同一基础模型，差异仅在于是否启用外挂。

## 四、数据安全与权限边界

**核心判断**：客户数据入表前先脱敏、入表后按字段授权，是"数据不出机构飞书租户"这一承诺在操作层面的两道闸门；两道闸门缺一不可——只脱敏不授权，租户内任何人可见；只授权不脱敏，一旦权限配置失误即造成泄露。

### 4.1 客户数据入表的脱敏规则

| 字段 | 处理方式 | 说明 |
|---|---|---|
| 客户姓名 | 不入表；以代号（如"客户 A"）替代 | 真实姓名留在机构自有的客户台账中，不进飞书 |
| 手机号 | 不入表 | 联系方式不进飞书 |
| 身份证号 | 不入表 | 证件信息不进飞书 |
| 持仓金额 | 区间化（如"100–300 万"） | 精确金额不入表，区间足够支撑适当性匹配 |
| 持仓明细 | 类别化（如"权益类为主"） | 具体标的与数量不入表 |

以上规则为 MVP 设计，具体执行粒度待与潜在用户机构确认后调整。

### 4.2 字段级权限与可见范围

| 表 | 顾问（普通使用者） | 合规/管理人员 |
|---|---|---|
| 资讯研报库 | 可读可写 | 可读可写 |
| 观点库 | 可读可写 | 可读可写，可标记失效 |
| 产品库 | 只读 | 可读可写 |
| 客户库 | 仅可见自己负责的客户记录 | 全量可见，可配置权限 |

多维表格的字段级与记录级权限能力以企业版为前提，具体套餐要求**待实测确认**；完整的五表×三角色权限矩阵与备用方案见 `docs/08_产品与客户数据模型.md` §七。

### 4.3 与现行监管要求的对应

下表与 `docs/01_开题方案.md` §十六 的口径保持一致，仅补充飞书载体下的落点：

| 监管文件 | 原型中的对应处理 | 飞书载体下的落点 |
|---|---|---|
| 《证券期货投资者适当性管理办法》（证监会令第 130 号） | 读取客户画像，匹配产品风险等级，记录风险揭示；不向风险不匹配的投资者主动推介 | 客户库存风险等级字段，产品库存产品风险等级字段，匹配与揭示记录写入审批流 |
| 《生成式人工智能服务管理暂行办法》（网信办等七部门令第 15 号） | 保护用户输入，标识 AI 生成内容，为备案与安全评估预留材料 | AI 生成内容在消息卡片与话术中显式标识；输入数据按 §4.1 脱敏 |
| 《证券期货业网络和信息安全管理办法》（证监会令第 218 号） | 数据分类分级、按角色授权、脱敏与委托处理边界 | §4.1 脱敏规则 + §4.2 字段级权限 + §3.3 外挂部署边界讨论 |
