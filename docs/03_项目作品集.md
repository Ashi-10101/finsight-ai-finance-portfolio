## 二、金策 FinSight：方案概览

金策是一套借助飞书 AI 工具、面向中小财富机构（中小 B）顾问及其高净值客户（大 C）的智能投研与财富服务助手，目前处于**方案设计 + 产品原型阶段，未开发完成、未上线**。

其服务链路为：资讯与研报入库（登记来源、时间与原文锚点）→ 观点生成（事实/共识/推断分层、登记失效条件）→ 产品影响判断（在售产品要素穿透）→ 客户分层（脱敏画像，风险等级 × 持仓）→ 分组话术生成 → 外挂机检拦截（可解释理由）→ 飞书审批流签发 → 留痕归档。系统承载于飞书：五张多维表格（资讯研报库 / 观点库 / 产品库 / 客户库 / 话术记录表）、知识问答、AI 字段捷径与 AI 工作流、审批流；飞书原生 AI 做不到的四项能力——证据字段级绑定、失效条件登记与到期重审、引用支持率校验、话术违规表述拦截——由自研外挂服务补齐。

在这套链路中，我负责产品与系统设计，重点处理三件事——也是我认为金策区别于普通"金融 GPT"的三个着力点：

1. **划分 6 类智能体职责及其分工**（资讯解读、观点整理、产品理解、客户匹配、话术生成、合规复核），但不把"大模型、多智能体、RAG"本身当作创新点；创新只可能来自分工是否让结果更稳、更可核对。
2. **设计证据链字段**，让每条结论都能回到来源、原文位置、关键假设、反例、置信度与失效条件，而不是只在文末附几条链接。
3. **将合规检查自动化前置并嵌入审批流**，从机制上避免系统退化为自动荐股或替客户决策的工具。

需要明确的边界：客户信息与持仓仅使用模拟数据；所有评测指标均为待验证目标；尚未在飞书上实际搭建任何对象，也未接入任何金仕达系统。未来若接入机构 CRM 或行情数据，也只在企业授权后以轻量集成方式进行。

## 四、Memomeld：知识管理与工作流

**起因。** 个人与小团队的资料常年分散在文档、笔记与对话之中，事后很难统一分类与检索。为解决这一问题，我开始做 Memomeld，计划以 Notes 为核心组织知识与任务。

**我做了什么。**
- 搭建 Node.js 全栈项目与 Docker Compose 环境，拆分 web 与 Postgres 服务
- 实现健康检查、环境模板与 Zod 校验
- 完成 Notes 模块的类型与校验层，并搭好数据库操作层结构
- 部署 memomeld-web 与 Postgres，并持续维护运行环境

**当前状态。** 项目处于**开发中**。工程骨架已完成并稳定运行 6 周以上，Notes 的核心业务逻辑仍在开发，因此不能表述为一个完整的知识库产品。后续计划先完成 Notes CRUD 与标签系统，再考虑多模型摘要、分类与知识图谱。

**与金策的关系。** 金策需要管理公告、研报、新闻、政策与内部资料。Memomeld 提供的是资料建模、校验与部署经验；面向金融的具体知识库尚未在金策中实现。

## 五、Mindraft AI Gateway：多智能体协作基础设施

**为什么做。** 单次模型调用难以稳定完成"拆解—执行—审核—整合"这类复杂任务。我把这套过程做成一个可查看执行状态与完整记录的服务，关键在于——它保留的不只是最终回答，而是整条执行轨迹。

**已实现。**
- Chat 提交任务后，Gateway 创建 run，并执行 Planner→Worker→Reviewer→Integrator 四阶段 Pipeline MoA
- 加入单轮 repair 与模型回退
- 通过 SSE 推送实时状态，在 Run Detail 展示 finalAnswer 与完整 trace
- 将 runs 与 run_steps 写入 Postgres
- 编写数据库自动备份脚本并经 cron 执行，使用 rclone 上传异地存储
- 部署 gateway-api、gateway-web 与 Postgres；配套 new-api 多模型 API 网关也在生产环境运行

技术栈：Next.js 15、Fastify、Drizzle、Postgres、SSE、Zod；monorepo 包含 `apps/web`、`apps/api`、`packages/shared` 与 `packages/moa-core`。

**当前状态与下一步。** Phase 1 MVP **已完成并上线自用**，生产环境稳定运行 3–4 周。并行 Worker、Role/Model 编辑器、Integrator 流式输出、多用户与文件上传仍在规划中。

需要如实指出：Gateway 目前运行的是**通用**四阶段工作流，金策计划中的 11 个金融角色尚未实现。它是金策可以站上去的编排起点，但两者之间隔着"数据时效、计算口径、合规约束"这三重金融特有的要求，后续需基于 `moa-core` 扩展。

## 六、Mindraft OS：个人数字工作台

我把日历、项目、服务状态、常用入口与期货计算工具整合进一个工作台，以减少在多个应用之间反复切换的成本。

**已实现并日用。**
- Today、Launchpad、Calendar、Services、Projects、Growth Engine、Futures 等模块
- Google Calendar 实时接入
- Services 健康监控，明确区分 live、stale、unknown 三态；缺失遥测不被显示为健康
- Futures 期货仓位计算器，处理多空方向、合约 tick、手续费、风险、盈亏与盈亏比
- `⌘K` 全局搜索，以及桌面与移动端适配

项目使用 Next.js App Router、Node 22 与服务端健康适配器，已部署生产环境并保持 healthy。配套的 mindraft-home、mindraft-drive、ingest-ui 与 ingest-bot 也在运行。其中 AI Studio **仅为界面预览**，不表示已经接入真实 agent 遥测。

**与金策的关系。** 这个项目最有价值的迁移经验有两点：一是高密度工作台的信息组织方式，二是"对数据状态与来源逐项标记"的处理习惯——后者正是金策把"live/stale/unknown"思路延伸到"事实/推断/失效"分层的直接来源。

## 七、金融信息与交易工具

**Trader Terminal。** 我用状态机限制交易记录的修改过程，从机制上避免事后改写原计划，这与金策"结论一旦确认即留痕、不可静默改写"的思路一脉相承。
- 实现 DRAFT→PLANNED→ACTIVE→CLOSED→JOURNALED 状态流转
- 在 PLANNED 后永久锁定关键计划字段，在 JOURNALED 后将记录设为只读
- 实现 Checklist 双端校验
- 计算计划执行率，并对比守纪与违规交易的胜率
- 加入经济日历

技术栈为 Next.js 14、tRPC、Drizzle、SQLite 与 Zod。v0.1 **已完成**，当前为本地单用户版本。状态流转、字段锁定与过程留痕的做法，可直接用于参考金策的审核与审计流程。

**news-backend。** 维护生产环境中的 Python 管线，处理多源资讯聚合、事件聚合、每日简报与摘要、经济日历抓取、热点外部数据抓取与 AI 分发。该项目提供了金融事件与资讯管线的实际运维经验。

**TheFirstTrader 与 OpenBB 探索。**
- TheFirstTrader：搭建 Phase 1 工程骨架与 Trades 模块，项目开发中；前后端与 Postgres 已部署并健康运行 6 周以上
- openbb-workspace-backend：探索 OpenBB 金融数据工作区后端对接，目前处于原型阶段
