【第2页】金策 FinSight 项目简介

**本页结论：金策是先导项目能力在金融决策场景的自然汇聚，而非凭空的新项目。**

- 交付物：24页开题方案 / 21页行业竞品研究 / 6张高保真原型 / 8张架构图 / 评测方案 / 7份样本

【第4-5页】项目一：Memomeld — AI知识管理与工作流

**本页结论：Memomeld验证了"复杂信息的结构化沉淀"能力，这正是金融公告/研报/政策知识管理的前提。**

1. **要解决的问题**：个人与小团队的知识分散在文档、笔记与对话中，难以沉淀、分类与复用
2. **我的角色**：独立产品与工程负责人
3. **产品方案**：知识库+标签体系+任务工作流，Notes模块为核心载体
4. **技术方案**：Node.js全栈、Docker Compose（web+Postgres）、Zod全链路校验、模块化后端（types/schema/repository/service分层）
5. **已完成内容**：工程骨架（容器化、健康检查、环境模板）、Notes模块完整类型定义与校验层、数据库操作层结构、生产部署（稳定运行6周+）
6. **当前状态**：开发中（核心笔记业务逻辑实现中）
7. **后续计划**：完成Notes CRUD与标签系统→接入多模型摘要与分类→知识图谱化
8. **对金策的先导作用**：金融公告、研报、新闻、政策与内部资料的知识管理、分类与沉淀经验

【第6-7页】项目二：Mindraft AI Gateway — 多智能体协作基础设施

**本页结论：金策的"多智能体投研引擎"不是纸面概念——其四阶段编排、执行追踪与模型回退已在Gateway项目中真实运行。**

1. **要解决的问题**：复杂AI任务无法由单次调用可靠完成，需要任务拆解、角色分工、审核与整合的工程化编排
2. **我的角色**：独立架构师与开发者
3. **产品方案**：用户只描述需求，系统理解任务、组建AI团队、执行Pipeline MoA、产出最终结果；Chat提交→Run创建→四阶段执行→SSE实时状态→Run Detail查看finalAnswer+完整trace
4. **技术方案**：Next.js 15 + Fastify + Drizzle + Postgres + SSE + zod；monorepo（apps/web、apps/api、packages/shared、packages/moa-core）；共享DTO与事件契约；单轮repair机制；模型回退
5. **已完成内容**：Phase 1 MVP全部范围（Chat页、MoA四阶段、SSE状态、Run Detail、runs/run_steps落库）；零依赖数据库备份脚本+cron定时备份+异地rclone上传；生产部署稳定运行3-4周；配套new-api模型网关在生产运行
6. **当前状态**：Phase 1 MVP已完成并上线自用
7. **后续计划（规划中）**：并行Worker、Role/Model编辑器、Integrator流式输出、多用户、文件上传
8. **对金策的先导作用**：多智能体投研编排、模型路由、执行追踪、错误回退的直接技术基础；金策11角色工作流是MoA四阶段在金融领域的扩展

【第8-9页】项目三：Mindraft OS — 个人数字工作台

**本页结论：高密度、多模块、明确区分数据可信度的个人操作系统，为金融专业工作台的信息架构提供了真实样本。**

1. **要解决的问题**：任务、日历、知识、项目、基础设施状态分散在多个工具，缺乏统一的私人工作台
2. **我的角色**：独立产品与工程负责人
3. **产品方案**：模块化工作台（Today/Launchpad/Calendar/Services/Projects/Growth Engine/Futures/AI Studio预览），⌘K全局搜索，桌面+移动端适配
4. **技术方案**：Next.js App Router、Node 22、服务端健康适配器（明确区分live/stale/unknown数据provenance——"缺失的遥测永远不显示为健康"）、Google Calendar OAuth实时打通
5. **已完成内容**：上述多模块日用；Futures期货仓位计算器（合约感知的仓位、风险、盈亏、tick、手续费、盈亏比，多空双向）；生产部署healthy
6. **当前状态**：多模块已完成并日用；**AI Studio为明确标注的界面预览，不宣称真实agent遥测**
7. **后续计划**：模块间数据联动、移动端完善
8. **对金策的先导作用**：研究员/投资经理/财富顾问工作台的信息架构与交互设计经验；数据provenance分层思路直接启发了金策"事实/观点/推断分层"

【第10-11页】项目四：金融信息终端与交易工具链

**本页结论：从行情资讯聚合到交易纪律工程，参赛者以真实用户身份理解金融信息的生产、消费与风控。**

**4a. Trader Terminal（v0.1已完成，本地单用户）**
- 问题：交易执行缺乏纪律约束与可量化复盘
- 方案：DRAFT→PLANNED→ACTIVE→CLOSED→JOURNALED状态机；PLANNED后关键字段永久锁定；Checklist双端校验；核心指标=计划执行率、守纪vs违规胜率对比；经济日历
- 技术：Next.js 14 + tRPC + Drizzle + SQLite + Zod
- 先导作用：状态机式工作流与"过程留痕"意识→金策审计追溯与人工确认节点设计

**4b. TheFirstTrader（开发中，已部署prod健康运行6周+）**
- 交易记录、复盘分析、策略追踪；Phase 1工程骨架+Trades模块占位

**4c. news-backend资讯管线（运行中，个人使用）**
- Python多源资讯聚合、事件聚合器、每日简报/摘要生成、经济日历抓取、热点外部数据抓取
- 先导作用：宏观资讯聚合与"事件→简报"自动化→金策数据接入层场景经验

**4d. openbb-workspace-backend（原型阶段）**
- OpenBB金融数据工作区后端对接探索
