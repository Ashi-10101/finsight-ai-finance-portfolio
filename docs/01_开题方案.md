## 一、执行摘要

**概括**：金策是一套面向中小 B（区域券商分支、独立财富工作室、三方销售、保险经代、家族办公室、私募 FOF）与大 C（高净值与资深个人投资者）的智能化金融投研与财富助手，借助飞书 AI 工具落地；它把"每一条结论绑定到可核对的原始材料、可失效的时间条件和明确的责任人"作为第一性要求，先在单一宏观场景上把流程完整跑通，再谈规模化。

金策的主用户是中小 B 的财富顾问，同时服务机构负责人与合规兼岗人员，并以只读方式触达客户（大 C）；完整的用户定义见 §五。系统的技术载体是飞书：飞书多维表格、飞书知识问答、飞书多维表格 AI 字段捷径、飞书 AI 工作流与飞书审批流，架构映射、"零部署"的精确边界与"数据不出机构飞书租户"的口径详见 `docs/07_飞书实现方案.md`（飞书具体功能的可用性待实测确认）。其基本运行逻辑是：用户提出研究问题后，系统将其拆解为可并行的子任务，检索并**保存**原始材料（而非只保存摘要），由分工明确的多个智能体分别完成分析与交叉复核，再把宏观层面的结论逐层映射到具体产品与客户约束上；任何对外材料，都必须先通过合规检查，再经人工确认，才允许发布。

- **项目背景**：市场规模与技术成熟度之间存在明显落差。招商银行与贝恩《2023中国私人财富报告》显示，中国个人可投资资产在2022年末已达278万亿元，财富管理与投研服务的需求基数庞大；然而在专业金融问答基准 FinanceBench（arXiv:2311.11944）中，即便是 GPT-4-Turbo 叠加检索增强（RAG），面对150道以真实财报为依据的问题，仍有约81%的回答出现错误或直接拒答。一端是巨大的存量需求，另一端是尚不可靠的取证能力。这条落差正是金策的立足点。
- **项目范例**：全文以"央行宣布降准 0.5 个百分点"为贯穿案例：一家 6 位顾问、400 位客户（其中 180 位持有固收+产品）、在售 8 只产品的区域财富工作室，要在开盘后 1 小时内判断这条消息对 8 只产品的影响，按风险等级与持仓把客户分 4 组，并为每组产出一段已过合规审批、可直接发出的话术。详细场景见 §七。
- **项目基础**：团队此前已独立完成知识管理、多模型网关、个人工作台与金融交易工具等先导项目，积累了多智能体编排、执行状态记录、全栈开发与生产部署的实际经验。需要说明的是，这些项目是能力储备，而非金策的旧版本或改名。
- **当前限制**：我们尚无任何企业内部数据，也没有在真实机构流程中得到的评测结果。全文涉及的客户资料与持仓均为模拟数据，所有量化指标一律为 MVP 目标值或待验证项，不代表已实现的成绩。
- **合规边界**：AI 在金策中只承担辅助研究与辅助决策的角色。系统不承诺收益、不自动荐股、不代替客户或持牌人员作出投资决定；投资建议的最终责任始终留在持牌机构与持牌人员一侧。

## 二、对金仕达命题的理解

一件新工具能否被真正采用，很大程度上取决于它能否嵌入使用者既有的工作环境。而中小 B 的处境是**没有系统可嵌入**：它们的困境不在集成难度，在于根本没有可集成的东西。所以"可嵌入"在这里的含义，就是嵌入他们已经在用的办公工具。本命题要解决的因此是这样一个问题：研究工具如何借助飞书，进入中小 B 已经存在的协作、留痕与合规环境。

公开资料显示，金仕达成立于1995年，脱胎于复旦大学，服务超过400家中资金融机构，业务横跨金融市场交易管理、金融风险管理，以及大宗商品与产业数字化。这些客户是已经运行着成套核心系统的持牌大机构；而本命题的最新要求面向中小 B 与大 C，二者处境截然不同：中小 B 没有投研团队、没有 IT 部门，也没有运行核心交易系统，但它们对"数据可控""操作可审计""责任可追溯"的要求并不更低，只是满足这些要求的方式不可能是私有化部署与系统集成，只能是依托其日常办公工具。基于此，我们把金策定位为"借助飞书 AI 工具、可零部署进入中小 B 的财富助手"，而非"另起炉灶的独立平台"；"零部署"的精确边界与"数据不出机构飞书租户"的口径，见 `docs/07_飞书实现方案.md` §一。

下表基于金仕达公开产品信息，梳理其能力与金策的结合位置。需强调，这些均为**面向未来的落地设想**，当前不存在任何实际系统对接，全部以企业授权为前提。

| 金仕达公开能力 | 金策设想中的结合位置 |
|---|---|
| 帝喾大模型 | 设想中：在因果分析方向探索能力协同，对应金策的市场观点整理环节 |
| 合规大模型 | 设想中：对接法规知识库与合规审查能力，对应 `docs/07_飞书实现方案.md` §3.2 的话术违规表述拦截外挂 |
| 易经金综合金融终端、智能策略交易系统 | 设想中：若机构已使用该终端，可作为研究入口的补充载体 |
| 期货交易管理系统 V6.0、V8T、DTP-F 核心交易系统 | 中小 B 大多不运行核心交易系统，属大 B 场景的远期可能（见 §二十） |
| KGo 多元资管系统 | 同上：大 B 场景远期可能，中小 B 场景不作前置依赖 |
| CTRM 大宗商品贸易与风险管理系统、HMS 套期保值管理系统 | 面向产业客户，与本方案的中小 B 财富管理场景不重合，不纳入 |

我们对命题的理解可以概括为一句话：**"可信"比"更强"更稀缺，"可嵌入"比"更全能"更重要。** 在中小 B 场景下，"可嵌入"的具体答案就是飞书。本项目将把有限的资源集中投向前者。

## 三、行业背景与需求变化

金融机构对大模型的态度，正从"要不要用"转向"如何用得可控"。行业当前真正稀缺的，不是又一个能写研报的模型，而是一个材料可核对、权限可管理、责任可界定的可信投研助手。

支撑这一判断的几条公开事实如下：

- **需求侧规模庞大**。中国个人可投资资产在2022年末为278万亿元（招商银行×贝恩《2023中国私人财富报告》）；麦肯锡曾预计中国个人金融资产将在2025年达到约332万亿元。庞大的资产存量对应着持续的配置、研究与顾问需求，这是投研工具的市场底座。
- **服务模式正在迁移**。财富管理从一次性"卖产品"转向持续性"做顾问"，行业内"三分投、七分顾"的说法反映的正是这一重心转移；基金投顾试点自2019年10月启动。（关于获批机构数量，行业报告中"约60家"是阶段性统计口径，本项目未重新核验，仅用于说明趋势。）服务模式越"顾问化"，对分析过程的可解释性与留痕要求就越高，这正是金策试图补足的部分。
- **供给侧竞相入场**。同花顺 HithinkGPT、恒生电子 LightGPT 及"光子"、东方财富"妙想"、蚂蚁 AntFinGLM、度小满"轩辕"等金融大模型或应用先后发布。2025年2月前后，多家媒体报道券商开始本地化部署 DeepSeek-R1/V3，部分报道统计为"至少16家"。（该数字受统计时点与口径影响较大，本文仅用它说明一个方向性事实）因此我们得出结论：**私有化、本地化部署正在扩散**，机构越来越倾向于把模型放进自己可控的环境里，这与金策"数据不出域"的设计取向一致。
- **各大投资研究机构需求迫切**。麦肯锡全球研究院曾估计知识型员工约19%的工作时间用于检索信息。

据上述事实我们团队得出：当"能不能生成"不再是问题，机构关心的焦点自然下移到以下三点：**材料能否被核对、数据权限如何处理、对外内容由谁负责**。这正是本项目选择"从研究流程切入"而非"从单一模型切入"的根本原因：模型是可替换的部件，而流程、留痕与责任边界才是机构真正难以自行补齐、也最愿意为之付费的部分。

## 四、问题从哪里来

在金融投研里，算出答案往往是最容易的一步。真正难处理的是三件事：这个答案的**出处**在哪、**时点**是否还成立、出了错谁负**责任**。通用大模型最容易忽略的恰好就是这三个维度。

需要坦诚说明：本项目尚未开展正式的用户访谈。以下五点判断来自公开研究、竞品观察，以及本人长期使用行情、宏观资讯与交易复盘工具过程中反复遇到的真实困扰。我们把它们列为待验证的问题假设，而非既成结论。

1. **长材料中的取证并不稳定。** FinanceBench 的结果（150题中约81%错误或拒答）指向一个结构性问题：模型在面对动辄上百页的财报、纪要时，难以稳定地"定位到正确的那一句话再作答"。错误往往不是算术错误，而是取证错误，引用的是看似相关、实则不支持结论的段落。对投研而言，取证失败比不作答更危险，因为它披着"有依据"的外衣。
2. **金融数字具有强烈的时间属性。** 利率、库存、财报指标、政策表述都会被后续数据覆盖。一个不显示"截至何时有效"的答案，使用者无法判断它是仍然成立还是早已过期。金策因此把"时点"提升为一等字段：任何数字都必须携带发布时间与更新时间。
3. **观点极易被写成事实。** 这一条我自己吃过亏。看资讯的时候，"市场普遍认为"这五个字很容易就被当成既成事实读过去；等真去追这句话的出处，常常发现它起初只是某一家机构的观点，经过几轮转述，主语掉了，读起来就成了共识。市场共识、争议意见、模型推断与情景假设一旦混在同一段流畅的文字里，报告读起来更顺，误用风险却更高。这要求系统在生成阶段就把客观事实与主观判断分层标注，并记住每条观点最初是谁说的，而不是事后靠读者自行分辨。
4. **市场判断不能直接跳到客户建议。** 同一个宏观结论，对不同持仓结构、不同投资期限、不同风险等级的客户，含义可能完全相反。省略"经过组合与客户约束这一层"的建议，本质上是把市场观点冒充为投资建议。
5. **机构要为输出结果承担责任。** Air Canada 案（2024 BCCRT 149）中，航空公司被裁定需为其聊天机器人提供的错误信息负责，这一判例说明"是AI说的"并不能免除机构责任。放到金融场景，还要叠加投资者适当性、数据安全与特许经营边界等更严格的约束。责任无法外包给模型，系统就必须让责任可追溯、可界定。

这五点之间存在内在关联：取证不稳（1）与时点缺失（2）损害的是**可核对性**，观点混淆（3）与越过客户约束（4）损害的是**可解释性**，而责任归属（5）则决定了前四点为什么非解决不可。金策的全部设计，都是围绕这三条主线展开的。

## 五、目标用户

**核心判断**：金策的主用户是中小 B 的财富顾问。他不缺信息，缺的是把一条资讯在一小时内变成 N 段能对不同客户说、且敢发出去的话。

为什么先切中小 B 而不是大 B：持牌大机构有投研团队、有自建系统，AI 对它们是既有流程上的改善；中小 B（区域券商分支、独立财富工作室、三方销售、保险经代、家族办公室、私募 FOF）两者都没有，同样的能力带来的是从 0 到 1 的变化【推断】。四类角色如下。

| 角色 | 日常处境 | 系统提供什么 | 不提供什么 |
|---|---|---|---|
| 财富顾问（主用户，系统的实际操作者） | 研究由本人兼职完成；客户分布在不同风险等级，同一事件要产出多个版本的沟通内容 | 资讯解读、研报摘要、市场观点整理、产品×客户匹配、分层话术草稿与风险提示 | 个股买卖指令、收益承诺、未经确认的内容自动外发 |
| 机构负责人 | 关心团队产出的一致性与机构整体风险敞口，本人不高频使用 | 审批流中的签发权、团队话术口径的查看与统计 | 绕过审批直接对外发布的通道 |
| 合规兼岗人员 | 中小 B 通常**没有专职合规**，多由业务或行政人员兼任，无法逐字把关全部对外内容 | 机检前置：违规表述拦截、风险提示完整性校验在送审前自动完成，并给出可解释的拦截理由 | "全靠人工逐字把关"的流程——那在兼岗现实下等于没有把关 |
| 客户（大 C，高净值与资深个人投资者） | 在群或一对一沟通中接收顾问发来的内容 | **只读接收方，不是系统操作者**：收到的是经持牌顾问确认后的材料 | 系统的任何操作入口；模型的原始输出 |

其中"合规兼岗"这一行需要展开：正因为中小 B 没有专职合规，金策才把合规检查设计为**自动化前置**：能在机器层拦下的（收益承诺、保本稳赚表述、风险提示缺失），绝不留到人工环节；兼岗人员有限的精力，只花在真正需要人工判断的项目上。这一设计对应 `docs/07_飞书实现方案.md` §2.6 与 §3.2 外挂四。

需要把 B2B2C 从"间接受益"升为本方案的**主架构**：AI 只对顾问说话，顾问对客户说话。这既是产品选择，也是合规必需。投资咨询在我国属于特许业务，大 C 拿到的永远是经持牌顾问确认后的内容，而非模型原始输出；这是唯一站得住的架构。

最后明确一条红线：任何人——包括顾问本人——都无法通过本系统获得个股买卖指令。这是**能力边界**，不是用户筛选；它不排斥任何一类使用者，只约束系统自己的能力范围。

## 六、一次任务如何流转

内部理解与对外话术应当共用同一批经过核对的材料，但两者的审核门槛必须不同：内部理解追求完整，对外话术追求适当、合规与可理解。在中小 B 场景下，这道区分不是更弱了，而是更重要了。

中小 B 不存在研究与销售的分工。如 §五 所述，顾问一人身兼研究、销售、合规三职，**一个人走完全程**。因此任务流设计为一条主线上的**两道门槛**：第一道门后是内部理解（顾问自己看），第二道门后是对外话术（客户看）。正因为写的人和发的人是同一个人，"把内部推断直接搬给客户"的风险反而更高。两道门槛不是冗余，而是唯一的防线；这也正是"机检必须前置"的根本原因。

以 §七 的降准场景为例，一次完整任务的主线如下：

| 步骤 | 动作 | 门槛 |
|---|---|---|
| 1 | 资讯入库：降准消息进入资讯研报库，登记来源、发布时间与原文位置 | |
| 2 | 观点生成：系统生成带原文锚点的市场观点，事实、共识与推断分层标注 | **第一道门**：经顾问确认，成为内部理解 |
| 3 | 产品影响：把观点映射到公司在售产品，逐只说明影响路径 | |
| 4 | 客户分层：按风险等级与持仓把客户分组 | |
| 5 | 分组话术：为每组生成话术草稿，话术绑定所依据的观点条目 | |
| 6 | 机检拦截：违规表述拦截与风险提示完整性校验自动前置 | |
| 7 | 审批：话术进入飞书审批流，由持牌顾问逐组确认 | **第二道门**：过门后才允许对外 |
| 8 | 分组发送与留痕：确认后发出，全程保留记录 | |

两道门槛共享同一份证据库，却在审核环节分流。这一设计的意义在于：它从流程上杜绝了"把内部推断直接搬给客户"的常见风险，同时避免了同一批材料被重复检索与重复核对。

## 七、原型场景：聚焦单一宏观事件

与其把多个场景铺开做成"演示拼盘"，不如选一个足够典型的场景，把每一步真正接起来。单一场景更能暴露流程断点。

**演示任务**：周一 9:05 央行宣布降准 0.5 个百分点。某区域财富工作室 **6 位顾问、400 位客户，其中 180 位持有固收+产品，公司在售 8 只产品**。要在开盘后 **1 小时内**：判断这条消息对这 8 只产品意味着什么 → 按风险等级与持仓把客户分 4 组 → 每组产出一段能直接发出去、且已过合规审批的话术。客户信息与持仓全部为模拟数据，界面持续标注。

| 步骤 | 原型中的实际动作 |
|---|---|
| 1 | 降准消息进入资讯研报库，登记来源、发布时间与原文位置 |
| 2 | 生成带锚点的市场观点，分层标注事实、市场共识与模型推断 |
| 3 | 把观点映射到 8 只在售产品，逐只说明影响路径与不确定性 |
| 4 | 按风险等级与持仓（180 位持有固收+产品）把 400 位客户分 4 组 |
| 5 | 为每组生成话术草稿，话术绑定所依据的观点条目 |
| 6 | 机检前置：违规表述拦截、风险提示完整性校验 |
| 7 | 话术进入飞书审批流，由持牌顾问逐组确认 |
| 8 | 确认后分组发出，留痕归档 |

选择这一场景的理由：它检验的是"一条资讯在 1 小时内走完到客户的全链路"——解读、观点、产品映射、客户分层、话术、合规、发送，任何一环断掉，1 小时的时限就守不住。也就是说，这一场景检验的维度是**链路能走多通**，而不是单一问题的分析深度。我们必须清醒：**场景跑通只证明流程可衔接，并不等于系统已经可用于真实业务**——后者需要真实数据、真实岗位与独立评测才能下结论。

## 八、产品边界

AI 可以承担投研中大量重复性劳动，但不能、也不应接走持牌人员的判断责任。金策的边界就划在"辅助"与"替代"之间。

金策是一套面向中小 B 财富顾问、并以只读方式服务其客户（大 C）的辅助研究与辅助决策系统。分工是明确的：**AI 负责检索、整理、计算、推演、起草与初步检查；人负责判断材料是否充分、结论是否成立、内容能否对外发布。** 下表通过与几类容易混淆的工具对比，界定金策的边界。

| 容易混淆的工具 | 金策的边界 |
|---|---|
| 财经聊天机器人 | 不以"有问必答"为目标；材料不足时明确提示缺失，而非编造补齐 |
| 新闻摘要工具 | 摘要只是起点，之后继续处理传导路径、情景推演与组合影响 |
| 自动研报生成器 | 中间材料与修改记录全程留存，报告只是流程末端的产物而非目的 |
| 自动荐股系统 | 不输出个股买卖指令，不承诺收益——任何使用者（含顾问本人与其客户）都无法从本系统获得此类指令 |
| 替代持牌投顾的自动决策工具 | 面向大 C 的任何输出必须经持牌顾问确认后发出，责任仍在持牌机构与人员一侧 |

此外，面向大 C 的任何输出，必须经持牌顾问在飞书审批流中确认后方可发出——审批流天然带责任人、时间戳与不可抵赖记录，这一映射的论证详见 `docs/07_飞书实现方案.md` §2.6（飞书审批的具体能力待实测确认）。

金策在被设计时加入了多智能体、检索增强与结构化数据处理，但这些都是成熟的实现方式，**本身不构成创新**。项目真正需要验证、也真正有价值的问题只有以下三点：1. 分工协作后是否比单模型更稳定？2. 引用是否真的支持了结论？3。 整套流程是否契合中小 B 机构的实际使用与合规要求？我们把这三点作为衡量项目成败的标尺，而不是罗列用了多少种技术。

## 九、产品模块设计

七个产品模块与命题要求的七项能力一一对应，构成一条"资讯→观点→产品→客户→话术→合规"的责任链，每一环都为下一环留下可追溯的输入。

| 命题能力 | 产品模块 | 飞书承载（详见 `docs/07_飞书实现方案.md`） |
|---|---|---|
| 资讯解读 + 研报摘要 | ① 资讯与研报接入 | 多维表格资讯研报库 + 飞书知识问答（待实测确认） |
| 市场观点整理 | ② 观点整理与分歧看板 | 多维表格观点库 + AI 字段捷径（待实测确认） |
| 产品理解 | ③ 产品库与产品理解（含持仓穿透） | 多维表格产品库 + 知识问答挂产品说明书（待实测确认） |
| 客户画像 | ④ 客户库与画像 | 多维表格客户库（脱敏规则见 `docs/07` §4.1） |
| （支撑全部模块） | ⑤ 证据管理 | 外挂一：证据字段级绑定（`docs/07` §3.2） |
| 风险提示 | ⑥ 风险提示与合规审核 | 飞书审批流 + 外挂四违规表述拦截（待实测确认） |
| 服务话术生成 | ⑦ 话术生成与分层触达 | AI 工作流 + 群机器人与消息卡片（待实测确认） |

两点设计取舍需要说明：

- **智能体协同分析不单独列为产品模块**：它由飞书 AI 工作流与外挂节点承载（`docs/07` §2.4、§3.3），属于实现方式。产品模块表只列"用户完成什么工作"，不列"怎么做"。
- **组合风险分析并入 ③ 的持仓穿透**：中小 B 顾问不做组合 VaR 与压力测试，他要回答的是"客户手上这只产品受什么影响"。完整的组合压力测试属更重的能力，非 MVP（见 §十五末）。

⑤ 证据管理保持原设计不动，九字段定义见 §十三。原型阶段将优先落地：四张多维表格、观点生成与锚点绑定、分组话术生成、机检拦截与审批流。技术实现层面的六层架构到飞书组件的映射见 `docs/07_飞书实现方案.md` §二，与产品模块并非一一对应，而是支撑关系。

## 十、用户界面组织

界面应围绕"任务如何推进"来组织，而不是把技术名词堆成功能列表。在飞书路线上，"界面"就是飞书内的原生载体：群消息卡片、多维表格视图、侧边栏与审批流。使用者关心的是"我现在该做什么、系统替我留下了什么"，而不是"这里用了什么算法"。

| 使用者在哪里操作 | 用什么飞书载体 | 系统留下什么痕迹 |
|---|---|---|
| 接收资讯解读、触发话术生成、一键送审 | 群消息卡片 + 群机器人（待实测确认） | 触发记录、生成版本、送审时间 |
| 维护与查阅资讯研报库 / 观点库 / 产品库 / 客户库 | 多维表格视图 | 字段级修改记录（待实测确认） |
| 查看客户画像与分组结果 | 侧边栏应用（待实测确认） | 查看与导出记录 |
| 送审、驳回、签发话术 | 飞书审批流 | 审批人、时间戳与审批意见 |
| 客户（大 C）接收内容 | 群消息（经顾问确认后由顾问发出） | 发送记录与内容版本 |

需要明确的是：「因果链画布」「情景比较」「证据工作区」这类重交互界面在飞书载体下不实现，**属暂不实现项，不属于 MVP**。它们承载的信息（传导路径、失效条件、原文锚点）由多维表格中的结构化字段与消息卡片中的只读展示承担。

## 十一、多智能体的分工机制

多智能体的价值不在"角色多"，而在分工之后能否互相制衡、让错误更早暴露。原型阶段先跑通最短路径，不一次性把所有角色堆齐。

智能体职责划分为 6 类：资讯解读、观点整理、产品理解、客户匹配、话术生成、合规复核——与 §九 的产品模块一一对应（证据管理贯穿其中）。（需要强调，这些名称只是**职责的划分**——实际部署时，既可以由不同模型分别承担，也可以由同一模型在相互隔离的上下文中分步执行，关键在于让"分析"与"复核"由不同的执行体完成，从而形成交叉检验。）

原型阶段先实现最短可用路径：

`资讯解读 → 观点整理 → 产品理解与客户匹配 → 话术生成 → 合规复核 → 人工确认（飞书审批流）`

编排由飞书 AI 工作流承担，证据绑定、引用校验与违规拦截由外挂节点完成（`docs/07_飞书实现方案.md` §2.4、§3.2、§3.3，飞书侧能力待实测确认）：AI 字段捷径承担单条记录内的文本变换，AI 工作流承担跨表流程。团队已有的 Mindraft AI Gateway 实现了 Planner→Worker→Reviewer→Integrator 四阶段管线，其编排经验——执行状态记录、失败回退、单轮修复——可参考，但飞书路线下不直接复用其代码。

异常处理遵循三条明确规则：其一，找不到材料就如实标注"缺失"，绝不用生成内容填补空白；其二，不同分析给出的结论**不取平均**，而是连同各自依据并列展示，把权衡留给人；其三，模型调用失败时可切换备用模型，但每一次切换都必须记录在案。凡涉及关键数据冲突、客户材料发布前、以及合规驳回后的处置，都强制要求人工介入。这三条规则的共同目标只有一个：**让系统在不确定时选择"停下并暴露"，而不是"流畅地蒙混过去"。**

## 十二、数据治理

数据在被模型使用之前，必须先回答清楚三个问题：来源何处、何时有效、面向对象是谁。顺序不能颠倒，否则可信与合规都无从谈起。

| 数据类别 | 内容 | 原型接入方式 |
|---|---|---|
| 市场数据 | 行情、宏观指标、利率与汇率 | 公开数据源；授权数据后续拟接入 |
| 基本面材料 | 财报、公告、招股书 | 公开披露渠道 |
| 文本资料 | 新闻、政策原文、研报、内部材料 | 公开来源；内部材料须经机构授权 |
| 客户数据 | 风险等级、投资期限、流动性约束、区间化持仓 | 原型阶段全部使用模拟数据；入表前按下表脱敏 |

客户数据入表前执行以下脱敏规则（与 `docs/07_飞书实现方案.md` §4.1 逐字一致）：

| 字段 | 处理方式 |
|---|---|
| 客户姓名 | 不入表；以代号（如"客户 A"）替代 |
| 手机号 | 不入表 |
| 身份证号 | 不入表 |
| 持仓金额 | 区间化（如"100–300 万"） |
| 持仓明细 | 类别化（如"权益类为主"） |

材料进入系统后，依次经过实体识别、来源与时间登记、有效性判断、冲突检测与原文定位五道处理。在检索技术选型上采用两路：**飞书知识问答的语义检索 + 多维表格结构化字段**（待实测确认），图谱检索不做。理由：同时维护三套索引对两人团队不现实，且中小 B 场景下"语义检索 + 结构化字段"已覆盖绝大多数查询（见 `docs/07` §2.3）。

在权限方面，凡涉及客户或内部数据，系统一律按角色授权，并在展示层执行脱敏，原始数据原则上**不出机构飞书租户**——口径与外挂部署形态的讨论见 `docs/07` §1.2 与 §3.3（MVP 默认采用形态三：外挂服务由金策统一托管、只处理已脱敏数据）。数据留在机构飞书租户内；私有化部署不在本方案覆盖范围内，具体见 `docs/07` §1.3。

## 十三、证据管理：一条结论需要保存什么

在报告末尾附几条参考文献是不够的。真正的可核对，要求在**形成结论的那一刻**就把所用材料绑定上去，并保留"它如何支持当前判断"的完整链条。

每一条重要结论，金策计划保存以下九个字段：

1. 来源机构、作者或系统内坐标
2. 发布时间与更新时间
3. 原文位置
4. 采用的事实依据
5. 从材料到结论的简要推理
6. 关键假设
7. 不支持该判断的反例或冲突材料
8. 置信度及其理由
9. 结论失效的条件

其中第7项（主动记录反例）与第9项（记录失效条件）是我们刻意强调的两点。绝大多数生成式工具只呈现"支持结论的证据"，而市场从来并非二元对立，同样一条新闻可能有不同解读，进而对市场价格造成截然相反的影响。金策要求同时保留"可能推翻结论的证据"以提供可能存在的风险依据，丰富决策时的信息背景。在界面上，已确认事实、市场共识、争议观点、模型推断、情景假设、行动建议与风险提示被明确分开；其中行动建议须经合规检查与人工确认，风险提示则设为**不可折叠**，防止在客户版中被弱化。

这一机制在复核环节的作用尤为关键：复核者可以逐条检查"引用是否真的支持了结论"。一旦发现无法支持，系统的正确反应是**删除或降级该表述**，而不是再生成一段"读起来更像答案"的文字。换句话说，我们宁可让报告显得保守，也不让它显得可信却错误。

## 十四、因果分析与情景推演

宏观传导路径应当被当作假设来对待：它们在特定条件下成立，条件一变就可能被推翻，没有哪条路径是可以照搬的公式。系统要做的，是让每一条路径都能被质疑、被重估。

以降准为例，原型会整理如下几条可能的传导路径，并明确它们各自的成立条件：

- **准备金率→银行可贷资金→市场利率→债券价格**：存款准备金率下调释放银行体系可贷资金，资金供给增加通常引导市场利率下行，存量债券价格相应上行，这是固收+产品受益的主要通道；
- **流动性宽松→风险偏好→权益资产**：流动性预期改善抬升风险偏好，对权益市场偏正面，但传导强度取决于企业盈利预期是否同步改善；
- **流动性宽松→货币市场利率→货币类产品收益**：货币市场利率中枢随资金宽松缓慢下移，货币类产品的收益率通常逐步走低。

但这些路径都不是唯一解。以权益市场为例，它同时受盈利预期、外部市场与汇率等因素驱动，单纯套用"降准利好股市"的路径会明显失真。因此，系统对每一环都要记录五项属性：**传导机制、影响方向、时间尺度、关键变量、失效条件**。举一个具体的反例以说明"失效条件"为何重要：常规判断是"降准→流动性宽松→债券受益"，但若宽松预期此前已被市场充分定价，降准落地反而可能"利好出尽"，债券收益率不降反升。这正是第 9 项字段（失效条件）在因果分析中的直接应用。

| 情景 | 示例假设 | 观察重点 |
|---|---|---|
| 宽松加码 | 降准后接续降息或其他宽松工具 | 债券延续走强、权益风险偏好修复 |
| 基准 | 降准落地，后续政策观望 | 债券小幅走强后震荡、权益内部分化 |
| 宽松低于预期 | 降准已被充分定价，无后续动作 | 利好出尽、债券收益率回升、权益回落 |

历史相似事件只能作为参照，且必须注明时间区间、样本数量与制度环境差异——2015年的降准周期与2020年的降准周期，其起点、节奏与外部环境都不相同，简单类比会误导判断。同样，情景发生的概率属于模型推断，绝不应被包装成确定性预测对外呈现。

## 十五、研究如何落到产品与客户

市场判断相同，客户手上的产品不同，该说的话就不同。从"市场观点"到"客户话术"之间，必须经过产品理解与适当性匹配这一层，不能跳过。

在中小 B 场景下，"研究落地"的对象不是组合，而是**产品 × 客户**。财富适配按以下五步进行：

1. **产品要素穿透**：逐只拆解在售产品的投向、久期、权益仓位与风险等级，回答"这条资讯对产品意味着什么"；
2. **客户分层规则**：按风险等级与持仓（区间化、类别化后的字段，见 §十二）把客户分组；
3. 比对产品风险等级与客户风险等级，执行《证券期货投资者适当性管理办法》（证监会令第130号）所要求的适当性检查——匹配对象是"产品 + 话术"，不向风险不匹配的客户发送对应话术；
4. 生成客户版材料，把事实、分析与建议分开呈现，并加入 AI 内容标识与风险提示；
5. 由财富顾问确认后方可对外使用，客户沟通全过程保留记录。

组合压力测试（集中度、风险敞口、情景映射与计算口径标注）在此明确标注为：**更重的能力，非 MVP，属大 B 场景的远期方向**。中小 B 场景下，顾问要回答的不是"组合在压力情景下变化多少"，而是"客户手上这只产品受什么影响"——前者需要完整持仓数据与确定性计算引擎，后者只需要产品要素与客户分层。

## 十六、监管要求如何进入流程

合规不应是发布前临时加盖的一道章，而应作为约束**内建**在数据、权限与内容生成的各个环节。检查必须发生在内容发布之前，且要覆盖数据与权限，而不只是措辞。

| 监管文件 | 原型中的对应处理 |
|---|---|
| 《证券期货投资者适当性管理办法》（证监会令第130号） | 读取客户画像，匹配产品风险等级，记录风险揭示；不向风险不匹配的投资者主动推介 |
| 《生成式人工智能服务管理暂行办法》（网信办等七部门令第15号） | 保护用户输入，标识 AI 生成内容，为备案与安全评估预留材料 |
| 《证券期货业网络和信息安全管理办法》（证监会令第218号） | 数据分类分级、按角色授权、脱敏与委托处理边界 |
| JR/T 0221—2021《人工智能算法金融应用评价规范》 | 保存模型调用、版本、输入输出与人工处置记录，设置模型回退与复核机制 |

在具体检查上，系统会针对四类高风险表述进行拦截：收益承诺、无依据建议、风险提示缺失、适当性冲突。凡涉及客户数据，一律执行最小授权原则，并完整保留操作日志、模型调用记录与提示词版本（提示词版本化的意义在于：当某条输出出现问题时，能够回溯到"当时是用哪一版指令生成的"）。需要特别界定的是：投资咨询在我国属于特许业务，金策不输出个股买卖指令，也绝不把 AI 生成的草稿当作"已经获批的投资建议"。

## 十七、产品原型页面

原型的目的是检验"信息是否够用、流程是否顺畅"，它不代表背后的模型能力已经完成。界面可用与结果可靠，是两件必须分开评判的事。

原型按 §十 的飞书载体组织，逐项说明"用户看到什么、系统留下什么"（飞书侧具体能力待实测确认）：

| 飞书载体 | 用户看到什么 | 系统留下什么 |
|---|---|---|
| 群消息卡片 | 资讯解读卡推送、观点草稿、话术草稿，附"送审"按钮 | 触发记录、生成版本、送审时间 |
| 多维表格视图 | 资讯研报库 / 观点库 / 产品库 / 客户库 / 话术记录表五张表（字段设计见 `docs/08`） | 字段级修改记录 |
| 侧边栏应用 | 客户画像面板：分组结果、风险等级与持仓类别 | 查看与导出记录 |
| 飞书审批流 | 话术送审单，附机检拦截结果与拦截理由 | 审批人、时间戳与审批意见 |

画布类重交互界面（因果链画布、情景比较等）**暂不实现**，其承载的信息由多维表格结构化字段与消息卡片只读展示承担（口径与 §十 一致）。

原型现状必须如实交代：现有 HTML 高保真原型正在按飞书载体重做，当前仓库中的 `prototype/` 目录内容尚未同步更新，不代表上述界面的最终形态。

原型评审真正要回答的问题是三个：顾问能否在**同一处**看到结论的依据？合规兼岗人员能否看清系统拦了什么、为什么拦？客户版话术是否成功避免了"把推断写成事实"。

## 十九、评测方案

评测的顺序本身就是一种价值主张：**先测引用是否可靠、数字是否正确，再谈生成有多快**。一个又快又错的系统，在金融场景里没有价值。

评测采用四组对照（完整设计见 `docs/06_评测方案.md`，实验尚未执行）：

| 组 | 配置 |
|---|---|
| A | 普通单模型 |
| B | 单模型 + RAG |
| C | 金策方案（飞书 AI 工作流编排 + 结构化证据链 + 产品与客户匹配 + 合规检查 + 四项自研外挂） |
| D | 飞书原生 AI 直接做（仅 AI 字段捷径 + 知识问答，不加任何自研外挂） |

预注册的主要比较为 **C 对 D**：两组共享同一批任务、同一份语料与同一基础模型，唯一差异是是否启用外挂——C 对 D 的差值用于归因证据绑定、失效重审、引用校验与违规拦截四项外挂的增益，回答"这些飞书自己不就能做吗"的质疑。内容质量类指标四组可比；效率与延迟类指标受平台差异影响，只在 C 与 D 之间比较（`docs/06` §2.1）。

主要指标与 MVP 目标值（待验证）：

| 指标 | MVP 目标值（待验证） |
|---|---|
| 事实准确率 | ≥ 95% |
| 数字准确率 | ≥ 98% |
| 引用支持率 | ≥ 90% |
| 无依据结论比例 | ≤ 5%（红线） |
| 信息时效性 | 时效合格率 ≥ 95%；时间戳字段完整率 = 100% |
| 产品要素抽取准确率 | ≥ 95% |
| 客户分层正确率 | ≥ 90% |
| 话术合规拦截率 | = 100%（红线）；误拦率 ≤ 10% |
| 话术可发率 | ≥ 60% |
| 合规风险识别 | 召回率 ≥ 90%；精确率 ≥ 85% |
| 单条资讯到全部客户触达的时长 | P50 ≤ 60 分钟 |

以上均为预期结果，待后续验证。

## 二十三、结语

**核心判断**：这个原型首先要证明的，不是"它能写出多漂亮的报告"，而是"当证据不足时，它会停下来"。

金融 AI 的风险，从来不只来自模型算错，更来自一段流畅的文字悄悄掩盖了三件事：材料的缺失、时点的错位、责任的模糊。金策的全部设计，都是从这三件具体的事出发的——保存原始材料以对抗"材料缺失"，区分事实与推断以对抗"责任模糊"，记录结论成立的条件以对抗"时点错位"，再把组合分析与发布前检查依次接上，形成一条可追溯的责任链。

衡量金策是否值得继续走向机构测试，标准同样朴素：它能否让研究员更快地找到结论的依据？能否让合规人员清楚地看见内容是如何形成的？以及——在材料不足时，它能否克制住"给出一个确定答案"的冲动，转而如实承认"证据不够"？如果这三个问题的答案都是肯定的，它才真正具备进入机构场景接受检验的资格。
